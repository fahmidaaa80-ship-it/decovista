# Decovista — Project Instructions

## State Management
- **Riverpod** (`flutter_riverpod: ^3.3.1`) — the ONLY state management library. Never add another.
- Use `StateNotifierProvider` for mutable/async state (e.g., cart, auth).
- Use `FutureProvider` for one-shot async data fetching.
- Use `Provider` (Riverpod's built-in type, NOT the separate `provider` package) for derived/simple synchronous state.
- Use `StreamProvider` for real-time streams.
- Widgets must extend `ConsumerWidget` (not `StatelessWidget`) or `ConsumerStatefulWidget` to access `WidgetRef`.
- Use `ref.watch()` for reactivity, `ref.read()` for one-time actions.
- Do NOT use `setState` for global/app-level state.

## Architecture — Feature-first layered
```
lib/
  core/          — shared infra: constants/, core/theme/, routes/, services/, widgets/
  data/models/   — data models with .fromJson() factory constructors
  providers/     — all Riverpod providers (state logic)
  features/      — feature modules: auth/, home/, shop/, cart/, booking/, profile/, prebuilt_designs/, admin/
```
Each feature module contains `screens/` and `widgets/` subdirectories.

## Routing
- **go_router** with `Provider<GoRouter>` defined in `lib/core/routes/app_router.dart`.
- Auth redirect logic lives inside the `GoRouter` redirect callback.
- Routes defined declaratively under `routes:`.
- Use `context.go()` for top-level navigation, `context.push()` for stacked screens.
- Named routes with path parameters (e.g., `/product/:id`).
- Do NOT import screen files directly in widgets — use router.

## Backend
- **Supabase** (`supabase_flutter`) via `SupabaseService` in `lib/core/services/`.
- All raw Supabase queries go through `SupabaseService` methods.
- Do NOT call `Supabase.instance.client` directly from features — use the service layer or providers.

## UI Conventions
- Use `ConsumerWidget` (not `StatelessWidget`) for any widget reading providers.
- Private widget classes prefixed with `_` in the same file.
- Reusable widgets in `lib/core/widgets/`: `CustomButton`, `LoadingWidget`, `CustomErrorWidget`.
- Colors from `AppColors` constants (`lib/core/constants/`).
- Images via `CachedNetworkImage`.
- Use `AsyncValue.when(data:loading:error:)` for async provider UI in build methods.

## Naming
- Files: `snake_case` (e.g., `cart_provider.dart`, `product_detail_screen.dart`).
- Classes: `PascalCase`.
- Variables/methods: `camelCase`.
- Screen files: `*_screen.dart` inside `features/<name>/screens/`.
- Widget files: `*_widget.dart` inside `features/<name>/widgets/`.

## Key Dependencies (do NOT add new state mgmt libraries)
go_router, cached_network_image, carousel_slider, flutter_rating_bar, badges, shimmer, font_awesome_flutter, flutter_form_builder, intl, table_calendar, image_picker, flutter_bkash, uuid, url_launcher, shared_preferences, flutter_dotenv.

## UI — STRICTLY DO NOT CHANGE
- Do NOT modify any existing UI theme, colors, typography, spacing, or layout.
- Do NOT restructure/reorder widgets in any screen.
- Do NOT change widget classes, widget trees, or visual hierarchy.
- Do NOT add/remove/modify any `MediaQuery`, `LayoutBuilder`, `Flex`, `Expanded`, padding, margin, alignment, or decoration values.
- Do NOT change `AppColors`, `AppTheme`, or any constant in `lib/core/`.
- Only add **logic** (event handlers, provider calls, navigation, data wiring) behind the existing UI.
- If a screen needs data that isn't wired yet, connect it via providers — never redesign the screen.

## Critical DOs and DON'Ts
- ✅ Use Riverpod providers for all shared state.
- ✅ Wrap app root with `ProviderScope` (already done in `main.dart`).
- ✅ Use `AsyncValue.when()` for async provider UI.
- ❌ Do NOT use `setState` for global/app-level state.
- ❌ Do NOT import providers directly from widgets in ways that break the pattern.
- ❌ Do NOT add any new state management library (no BLoC, GetX, Provider, etc.).
- ❌ Do NOT call Supabase directly from feature screens — use providers or SupabaseService.
- ❌ Do NOT place screen files outside `features/<name>/screens/`.
